# Data and Geospatial Design

> Status: Draft 0.1  
> Decision status: Approved baseline  
> Last updated: July 21, 2026

## 1. Purpose

This document defines the logical PostgreSQL/PostGIS design, ownership rules, identifier strategy, revision model, geospatial representation, auditing, deletion, and migration principles.

## 2. Authority

Cloud SQL for PostgreSQL is the synchronized source of truth. PostgreSQL owns accepted domain state, revisions, memberships, media metadata, job metadata, synchronization changes, and audit-relevant application facts.

Cloud Storage owns binary objects. Firebase Authentication owns credentials and external identity state. Neither replaces application domain records.

## 3. Schema Ownership

Use PostgreSQL schemas or naming conventions to make module ownership explicit. A module may query another module only through an approved public view, query port, or documented transaction use case.

Initial logical ownership is:

| Module | Data |
|---|---|
| Identity and Access | Profiles, identity links, account state |
| Collaboration | Gardens' membership, invitations, roles |
| Gardens and Mapping | Gardens, coordinate spaces, map objects, geometry revisions, calibration |
| Plants and Inventory | Plant instances, placements, garden-specific plant facts |
| Observations and History | Observations, measurements, history events |
| Tasks and Recommendations | Tasks, recommendations, evidence, outcomes |
| Media | Media records, variants, upload and retention state |
| Capture and Import | Capture sessions, imports, proposals, processor results |
| Platform | Idempotency records, outbox, sync changes, operational metadata |

## 4. Identifier Strategy

Use UUIDv7 identifiers generated by clients or trusted server components.

Properties:

- Identifiers are globally unique and opaque.
- Offline clients can create records without server allocation.
- Time ordering improves index locality but is not treated as authoritative creation time.
- External provider IDs are stored separately and never become primary keys.

## 5. Time and Actors

Every mutable aggregate records:

- Server creation time.
- Server update time.
- Current revision.
- Creating actor where relevant.
- Last modifying actor where relevant.

Client timestamps may be retained as observations but do not replace server ordering.

Actor references distinguish user, system worker, administrator, migration, and external import.

## 6. Garden Aggregate

A garden contains:

- Identity and ownership metadata.
- Locale, time zone, and optional location.
- Local coordinate space.
- Optional georeferencing transform.
- Map objects.
- Plant instances.
- Observations, tasks, and recommendations.
- Media and capture associations.
- Membership and role assignments.

The garden is an authorization boundary, not necessarily one transactionally locked aggregate. Object-level revisions prevent unnecessary whole-garden contention.

## 7. Garden Object Model

Use a hybrid model:

```text
garden_object
├── id
├── garden_id
├── category
├── geometry
├── coordinate_space_id
├── current_revision
├── provenance_id
├── confidence
├── lifecycle_state
└── common metadata

structure_details
fence_details
zone_details
bed_details
tree_details
plant_placement_details
```

Common spatial behavior remains queryable in `garden_object`. Specialized tables enforce category-specific fields and relationships.

## 8. Local Coordinate Space

Each garden has at least one local coordinate-space record:

- Identifier.
- Unit: meters.
- Axis convention.
- Origin description.
- Precision policy.
- Creation provenance.
- Lifecycle state.

Accepted editable geometry uses PostGIS `geometry` in the local planar space. The exact custom SRID registration strategy is decided in schema implementation, but the database and API must never label local coordinates as EPSG:4326.

## 9. Georeferencing

An optional georeference record maps local coordinates to WGS84 and contains:

- Garden and coordinate-space ID.
- Local anchor point.
- Geographic anchor point.
- Rotation.
- Scale correction if applicable.
- Accuracy or residual error.
- Provenance and method.
- Revision and validity interval.

Spatial queries that need geography transform explicitly into the approved geographic representation.

## 10. Geometry Types and Validation

Supported canonical types are initially Point, LineString, Polygon, and approved multi-geometries.

Database constraints and application validation enforce:

- Non-empty geometry.
- Expected geometry category.
- Valid polygons.
- Bounded coordinate magnitude.
- Finite numeric values.
- Coordinate-space consistency.
- Garden ownership consistency.

Cross-object semantic validation, such as plants inside blocked structures, occurs in application/domain services using PostGIS queries.

## 11. Measurements and Uncertainty

Measurements store canonical SI values and, where relevant:

- Original user-entered value and unit.
- Acquisition method.
- Accuracy or uncertainty.
- Reference object or segment.
- Calibration revision.

The schema must not imply survey accuracy merely because values use precise numeric types.

## 12. Provenance

Provenance records identify:

- Manual drawing.
- User-entered measurement.
- Imported plan or map.
- AR measurement.
- Video or image extraction.
- LiDAR or depth capture.
- External provider.
- AI or deterministic processor.

Generated results additionally reference processor version, model version, input media, and acceptance outcome.

## 13. Revision Model

Each mutable domain object has a monotonically increasing server revision within its identity. Updating an object:

1. Locks or conditionally updates the expected current revision.
2. Writes the new current state.
3. Writes an immutable revision snapshot or change representation.
4. Appends a synchronization change.
5. Appends relevant outbox events.

The revision journal supports history, conflict diagnostics, undo-as-new-change, and audit. It is not a full event-sourced rebuild requirement.

## 14. Append-Oriented Records

Observations and important history events are append-oriented. Corrections create a superseding record or explicit amendment when audit value matters.

Tasks and recommendations use explicit state transitions rather than overwriting history fields without an event record.

## 15. Deletion

Deletion states are explicit:

- Active.
- User-deleted and recoverable.
- Scheduled for purge.
- Purged.

Ordinary object deletion produces a tombstone for synchronization. Account and garden deletion start an asynchronous workflow that revokes access, handles shared ownership, deletes media, purges domain records in dependency order, and records policy-compliant completion evidence.

The baseline user recovery window is 30 days unless the user selects immediate irreversible deletion where supported or legal policy requires another behavior.

## 16. Synchronization Change Log

The server maintains an ordered change log containing:

- Global or partitioned monotonic sequence.
- Garden and record identifiers.
- Record type.
- Operation: upsert or delete.
- Accepted record revision.
- Authorization partition metadata.
- Commit time.

Payload strategy may use compact current snapshots or references. A client can resume from a durable cursor and receive deterministic bounded pages.

Change-log retention must exceed the supported offline interval or force a clearly handled full resynchronization.

## 17. Idempotency Records

Idempotency records are keyed by actor scope and idempotency key. They retain request fingerprint, operation, state, result reference, and expiration.

Retention is long enough to cover mobile retry and delayed synchronization behavior. Expiration cannot cause duplicate irreversible actions such as invitations, deletion, or billed processing without another guard.

## 18. Transactional Outbox

Outbox records contain:

- Event ID.
- Event type and version.
- Aggregate reference.
- Payload or payload reference.
- Creation time.
- Publication state and attempts.
- Trace context.

Domain commit and outbox insert occur in one transaction. A relay publishes to Pub/Sub and marks completion idempotently.

## 19. Media Metadata

Media tables store:

- Media ID and owner garden.
- Purpose and sensitivity class.
- Storage bucket and opaque object key.
- Content type and declared/verified size.
- Checksum.
- Upload and verification state.
- Original and derivative relationships.
- Retention and deletion state.
- Capture and provenance links.

Public or signed URLs are never stored as permanent identity.

## 20. External Reference Data

Plant taxonomy, weather observations, and provider content use normalized tables with:

- Provider and provider record ID.
- Fetch and effective time.
- License and attribution metadata.
- Raw payload reference only when retention is permitted.
- Normalized application fields.

User-authored garden facts remain distinct from provider facts.

## 21. Search

Initial search uses PostgreSQL:

- Full-text indexes for localized textual content where supported.
- Trigram indexes for names and tolerant matching.
- Structured filters for garden, category, lifecycle, and time.
- PostGIS indexes for viewport and spatial relationships.

Search documents are projections and can be rebuilt from authoritative tables.

## 22. Indexing

Baseline indexes include:

- Foreign-key indexes.
- Garden and lifecycle filters.
- Current-revision lookups.
- Outbox publication state.
- Sync sequence and authorization partition.
- Media processing state.
- Task due-state queries.
- GiST spatial indexes.
- Full-text and trigram indexes for approved search fields.

Indexes are justified by measured query plans. Redundant indexes are removed deliberately.

## 23. Partitioning and Archival

Do not partition domain tables initially without measured need. Candidates for later partitioning are high-volume append-only history, synchronization changes, telemetry-like operational records, and outbox archives.

Archival must preserve user export and deletion obligations.

## 24. Database Security

- Application and migration roles are separate.
- Runtime roles cannot create schema objects.
- Worker roles receive only required schemas and operations.
- No client receives a database credential.
- Private connectivity is required in staging and production.
- Sensitive columns are excluded from broad administrative views.
- Support access is time-limited and audited.

Application-level authorization is authoritative. PostgreSQL Row-Level Security may be introduced as defense in depth through a separate ADR after connection-pooling and administrative behavior are designed.

## 25. Migrations

- Migrations are reviewed SQL files with immutable identifiers.
- Migrations are forward-compatible with the currently deployed application during rolling releases.
- Destructive changes use expand, migrate, contract phases.
- Large backfills run as observable jobs, not one unbounded deployment transaction.
- PostGIS extension upgrades receive compatibility testing.
- Production migrations run through a controlled service identity and produce audit output.

## 26. Backup and Restore

Cloud SQL automated backups and point-in-time recovery protect the database. Restore tests verify:

- Extensions and spatial indexes.
- Application roles.
- Migration state.
- Synchronization sequence continuity.
- Media reference consistency.

Database restore does not automatically restore Cloud Storage objects, so recovery procedures reconcile both systems.

## 27. Testing

- Repository tests run against real PostgreSQL/PostGIS containers.
- Migration tests upgrade representative old schemas.
- Property tests cover geometry validity and transformation.
- Concurrency tests update the same expected revision.
- Sync tests cover tombstones, retention gaps, and cursor resume.
- Backup restoration is exercised in staging.
- Query-plan regression tests protect important spatial and list queries.

## 28. Completion Criteria

- Every authoritative record has clear module ownership.
- Offline identifiers can be accepted without remapping.
- Local and geographic coordinates cannot be confused.
- Geometry validation and indexes use PostGIS.
- Current state, history, sync changes, and outbox commit consistently.
- User deletion reaches media and derived data.
- Schema migration can occur without losing mobile compatibility.
